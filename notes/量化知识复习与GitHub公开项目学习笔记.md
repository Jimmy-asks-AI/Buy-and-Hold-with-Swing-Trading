# 量化知识复习与 GitHub 公开项目学习笔记

学习日期：2026-05-23  
对应代码：`strategy_lab/multi_factor_research_framework.py`

## 复习结论

已有资料库对量化研究的理解总体正确，但需要从“单个因子有效”升级到“可治理的大规模因子工厂”。数百因子模型的关键不是简单堆叠更多变量，而是建立：

- 因子注册表。
- 数据可得日期和未来函数审计。
- 横截面清洗、标准化和正交化。
- 单因子诊断。
- 冗余因子聚类。
- 因子家族内合成。
- 家族间模型合成。
- 组合权重和执行约束。
- 样本外、成本、容量、PBO 和实盘漂移监控。

这与已学习的卖方研报、论文和行为金融材料一致，也与 GitHub 上成熟量化开源项目的工程模式一致。

## GitHub 项目学习

### Microsoft Qlib

Qlib 的核心启发是端到端但松耦合：数据处理、模型训练、回测、风险模型、组合优化和订单执行是一条完整链，但每个组件可以独立替换。Qlib 还提供 Alpha158、Alpha360 等公开因子数据集思路，说明大规模因子库需要标准化数据处理和统一 workflow。

对本地框架的修正：

- 不能只写单因子函数，必须有因子注册表和统一 workflow。
- 模型输出要进入组合和执行层检验。
- 机器学习模型应支持监督学习、市场状态建模和强化学习，但必须共享同一数据和评估层。

来源：`https://github.com/microsoft/qlib`

### Quantopian Alphalens

Alphalens 的核心启发是因子诊断必须标准化：returns analysis、IC analysis、turnover analysis、grouped analysis 缺一不可。它证明单因子研究不应停留在“某个回测收益不错”，而应输出可比较的 tear sheet。

对本地框架的修正：

- 每个因子必须输出 RankIC、分组收益、换手、覆盖率和分行业/分组分析。
- 因子入库之前先做诊断，而不是直接进组合优化器。

来源：`https://github.com/quantopian/alphalens`

### VectorBT

VectorBT 的核心启发是大规模参数和因子扫描需要向量化、广播和快速组合分析。它适合做研究初筛和参数扫描，但不能替代真实执行模拟。

对本地框架的修正：

- 大规模因子库初筛可以向量化。
- 最终策略必须补充事件驱动回测、滑点、成交约束和成本。

来源：`https://github.com/polakowo/vectorbt`

### QuantConnect Lean

Lean 的核心启发是事件驱动和生产一致性。它把回测、实时数据、订单、经纪商接口、另类数据和模块化策略组件统一在一个工程框架内。

对本地框架的修正：

- 因子模型生成的不是最终收益，而是目标权重或订单意图。
- 组合构建后必须进入执行层检验。
- 回测和实盘最好共享同一套订单、费用和数据语义。

来源：`https://github.com/QuantConnect/Lean`

### Backtrader

Backtrader 的核心启发是交易细节显式化：数据源、时间框架、订单类型、手续费、滑点、成交量约束、分析器都应该是策略研究对象。

对本地框架的修正：

- 高频和中低频策略不能只看信号收益。
- 做 T、择时、轮动等策略必须明确成交价、成交量、费用和调仓方式。

来源：`https://github.com/mementum/backtrader`

### FinRL / FinRL-Trading

FinRL 强调金融强化学习的三层结构：市场环境、智能体、金融应用。FinRL-Trading 进一步强调以目标权重向量作为策略和执行之间的接口。

对本地框架的修正：

- RL 不能只优化 reward 曲线，必须有市场环境、交易成本、基准和稳健性测试。
- 大规模因子模型最终应输出目标权重，而不是直接输出“买卖建议”。

来源：

- `https://github.com/AI4Finance-Foundation/FinRL`
- `https://github.com/AI4Finance-Foundation/FinRL-Trading`

### FinGPT / FinRobot

FinGPT 和 FinRobot 的核心启发是 LLM 在金融中更适合做数据抽取、情绪理解、报告生成、RAG 和工具编排，而不是直接生成可交易 alpha。

对本地框架的修正：

- 文本和 LLM 因子必须有发布时间、来源、覆盖率和任务基准。
- LLM 产生的结论必须回到结构化数据和可重复回测。
- Agent 可以辅助研究，但不能跳过因子检验和组合检验。

来源：

- `https://github.com/AI4Finance-Foundation/FinGPT`
- `https://github.com/AI4Finance-Foundation/FinRobot`

### Microsoft RD-Agent

RD-Agent 的核心启发是自动化量化研发应是“提出假设 -> 实现代码 -> 运行实验 -> 读取反馈 -> 修正假设”的闭环。它还强调因子和模型联合优化，而不是单独堆因子或单独调模型。

对本地框架的修正：

- 每次自动研究都必须有复习和纠错。
- 因子挖掘和模型选择要联合，但必须防止过拟合。
- 研究代理要保留实验轨迹和失败样本。

来源：`https://github.com/microsoft/RD-Agent`

### QuantStats

QuantStats 的核心启发是绩效报告要模块化：统计指标、图表、报告、蒙特卡洛和尾部风险要分开。收益序列分析也要区分 period-based metrics 与 trade-based metrics。

对本地框架的修正：

- 组合收益评估不只看年化收益，还要看波动、Sortino、Calmar、最大回撤、胜率和尾部风险。
- 日频收益胜率不能直接等同于交易胜率。

来源：`https://github.com/ranaroussi/quantstats`

### pysystemtrade、RQAlpha、vn.py

这类系统的启发是量化研究最终要落到系统设计、风险预算、仓位管理、经纪商接口和生产运维。尤其是跨品种、期货、A 股实盘和套利场景，单纯研究脚本不够。

来源：

- `https://github.com/robcarver17/pysystemtrade`
- `https://github.com/ricequant/rqalpha`
- `https://github.com/vnpy/vnpy`

## 对已有知识的纠错

1. 因子数量不是护城河。数百因子只有在经济含义、可得日期、冗余控制和样本外稳定性都成立时才有价值。
2. 机器学习不是因子检验替代品。ML 只能学习非线性组合，不能修复脏数据、未来函数和过拟合。
3. 强化学习不是默认更高级。没有真实环境、费用、冲击、仓位限制和样本外评估的 RL 只是 reward 拟合。
4. LLM 不是直接交易员。LLM 更适合信息抽取、文本因子、研究助理和报告生成，交易信号必须结构化回测。
5. 快速向量化回测不是最终证据。它适合初筛，最终仍要事件驱动或执行层复核。
6. 因子合成不能直接全体平均。应先做家族内合成，再做家族间权重，避免某一类相似因子淹没其他信息。
7. 正交化不能滥用。正交化可以识别增量信息，但也可能剥离真实经济暴露；需要同时保留原始效果和正交效果。
8. 因子择时不是收益增强器。更准确地说，它是风格失效风险控制器。

## 数百因子模型的能力要求

一个合格的大规模因子模型应至少能处理：

- 价值、质量、盈利、成长、投资、杠杆、分红、估值修正。
- 动量、反转、波动、流动性、价量、技术指标。
- 高频、Level2、大单、LOB、日内结构。
- 分析师、机构、基金隐含、融资融券、薪酬、公告文本。
- 宏观敏感性、利率、汇率、信用、政策不确定性。
- 另类和准另类数据。
- AI 挖掘因子、深度时序因子、LLM 文本因子。

但它们必须被组织成家族，而不是平铺成一张巨大的特征表。平铺会导致相似因子重复计票、过拟合和风格暴露失控。
